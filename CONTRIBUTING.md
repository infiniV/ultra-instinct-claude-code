# Contributing

Thanks for helping make this resource better.

## Rules

1. **Every tip must follow the template.** No exceptions.

```markdown
### #XX.YY Tip Title

> **Level:** Beginner/Intermediate/Advanced/Expert | **Impact:** High/Medium/Low
> **Source:** [repo-name](link), [repo-name](link)

**Problem:** One sentence describing what goes wrong.

**Do this:**
\```
code or command example
\```

**Why:** One sentence explaining the benefit.
```

2. **No tips without source attribution.** Every tip must link to at least one public repo, blog post, or official doc where the technique originated or was validated.

3. **No tips requiring installation to understand.** The reader should be able to understand and apply every tip just by reading it. If a tip requires a tool, explain what the tool does inline.

4. **No duplicates.** Before submitting, search existing tips. If your tip overlaps with an existing one, improve the existing tip instead of adding a new one.

5. **PRs target specific topic files.** Don't submit a PR that touches every file. Each PR should add or improve tips in one section (e.g., `tips/06-prompting.md`).

## How to Submit

1. Fork the repo.
2. Create a branch: `tip/short-description`.
3. Add your tip to the appropriate file in `tips/`.
4. Ensure your tip follows the template above.
5. Open a PR with a brief explanation of why this tip is useful and where you learned it.

## Fixing Errors

If a tip is wrong, outdated, or has a broken link, open an issue or submit a fix directly. No template needed for corrections.
